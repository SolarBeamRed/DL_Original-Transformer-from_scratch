    "### Multihead Attention Class:\n",
    "<u>Purpose</u>: Receives static embeddings and generates contextual embeddings <br><br>\n",
    "Conceptually, multiple weight matrices are used for multihead attention. During Implementation however, single set of large Weight matrices are used, which basically represent the multiple different weight matrices\n",
    "concatenated together. This helps with simultaneous attention calculation<br><br>\n",
    "\n",
    "```view()``` function is used to return a new tensor with same values but different shape. This is used because dimensions change when going from single head to multiple heads in the ```split_heads()``` function<br><br>\n",
    "\n",
    "<u>Dimensions</u>:<br>\n",
    "- Input dimensions received by ```forward()```: ```[batch_size, seq_length, d_model]```\n",
    "- After ```split_heads()```: ```[batch_size, num_heads, seq_length, d_k]```<br>\n",
    "- Dimensions received by ```calculate_attention()```: Same as dimensions after split_heads<br>\n",
    "- After ```combine_heads()```: ```[batch_size, seq_length, d_model]```<br><br>\n",
    "  \n",
    "Final dimensions returned after calculating attention: Same as received during forward, ie ```[batch_size, seq_length, d_model]```\n",
    "\n",
    "___\n",
    "### Positional Feed Forward Class:\n",
    "Nothing special, just a normal Feed Forward block with ReLU activation after the first layer. Helps performance by introducing potentially necessary non-linearity<br><br>\n",
    "- Architecture: 2 layers\n",
    "- First layer structure: ```[d_model, d_ff]```\n",
    "- Second layer structure: ```[d_ff, d_model]``` \n",
    "- Activation: ReLU on first layer\n",
    "Second layer's size being d_model ensures that dimensionality remains same, so further operations such as Add and Normalize are still implementable with simplicity\n",
    "\n",
    "___\n",
    "\n",
    "### Positional Encoding Class:\n",
    "max_seq_length does NOT mean seq_length of batch or examples. It's an arbitrary value which basically says \"how many positions should I keep my pe calculated?\" (pe is the table of positional encoding values for each dimension and for each position).<br> \n",
    "When x is sent to get positional encoding added, only pe terms until x's length are chosen and added automatically<br> <br>\n",
    "Dimensions:\n",
    "- pe: ```[max_seq_length, d_model]```\n",
    "- position: ```[max_seq_length, 1]```\n",
    "- div_term: ```[d_model/2]``` (later broadcast to each position automatically)\n",
    "- final pe: ```[1, max_seq_length, d_model]```\n",
    "```pe``` is finally unsqueezed at axis=0 to make it compatible with batch calculations. The same pe can be broadcast accross all examples in the batch, since positional encoding terms to be added remains same\n",
    "\n",
    "___\n",
    "### Encoder Class:\n",
    "Simple encoder block with one Multihead Attention block and one Feed Forward block, with Add and Norm steps where necessary. Dropout is also used as a way to regularize model<br><br>\n",
    "Two different ```norm``` objects are used, since there is need to maintain two separate γ and β values<br><br>\n",
    "Dropout randomly turns off some dimensions of attention outputs and feed forward outputs during the \"Add\" step before normalization. This seems to help the model generalize better and prevent \"memorizing\" patterns instead of understanding<br><br>\n",
    "Queries, keys and values are all calculated from input in the Encoder block<br><br>\n",
    "Layer normalization is used, since BatchNorm takes even padded tokens into consideration while normalizing, which is not ideal. LayerNorm prevents padded tokens from affecting normalization of other token embeddings, which is preferred over BatchNorm's behavior\n",
    "\n",
    "___\n",
    "### Decoder Class:\n",
    "Decoder block containing one Masked Multihead Attention block, one Cross Attention block and one Feed Forward block, with respective Add and Norm steps where required. Dropout is used in similar fashion as in Encoder block<br><br>\n",
    "Masks:<br>\n",
    "```src_mask```: padding mask <br>\n",
    "```tgt_mask```: combination of causal mask(mask used in masked multihead/\"nopeak\" mask) and padding mask <br>\n",
    "Mask generation seen in the Transformer section <br><br>\n",
    "\n",
    "Cross attention block uses decoder's ```attn_outputs``` for Query, and ```enc_outputs``` for Key and Value embeddings\n",
    "\n",
    "___\n",
    "### Transformer class input and mask generation:\n",
    "Contains the whole architecture, including blocks for generating static embeddings for Encoder and Decoder, the Encoder-Decoder block themself and mask generation <br><br>\n",
    "\n",
    "Format:<br>\n",
    "- Inputs: ```src: [batch_size, src_seq_length]``` and ```tgt: [batch_size, tgt_seq_length]```\n",
    "- src_mask: ```[batch_size, 1, 1, seq_length]```\n",
    "- tgt_mask(before integrating with causal): ```[batch_size, 1, seq_length, 1`]```\n",
    "- causal_mask: ```[1, seq_length, seq_length]``` \n",
    "- Final tgt_mask: ```[batch_size, 1, seq_length, seq_length]``` after 'AND' operation with causal mask\n",
    "<br>\n",
    "Padding masks are both boolean, with 'True' values in non padded indices, and 'False' for padded token indices<br>\n",
    "Causal mask is a square matrix of [1, seq_length, seq_length] with upper triangle elements being 0s, and rest being 1s. <br>\n",
    "\n",
    " ```triu()``` with diagonal=1 returns a matrix with 1s in upper triangle, and 0s in remaining positions, which is then subtracted from 1 to get required causal mask<br>\n",
    "3*3 example:\n",

    "     [0 0 0]]         [1 1 1]]\n",
    "```\n",
    "\n",
    "Then integrated with padding mask using 'AND' operation<br><br>\n",
